## Проблема: Почему ценные знания теряются

В горно-металлургических исследованиях знания рассеяны по тысячам документов:

- **Потеря институциональной памяти** — методы обессоливания воды, режимы циркуляции католита, распределение металлов между штейном и шлаком погребены в отчётах и личных архивах
- **Дублирование усилий** — команды повторяют литературные обзоры, не видя, что работа уже выполнена
- **Междисциплинарная слепота** — невозможно быстро найти связь «режим кучного выщелачивания в холодном климате → выход металла» когда данные разрознены
- **Медленные решения** — ответ на вопрос «какие решения циркуляции электролита существуют?» требует ручного сбора из десятков источников
- **Риск противоречий** — без единой верифицированной базы множатся конфликты интерпретаций

**Nikel** — решение, которое превращает этот хаос в управляемую, семантически связанную карту знаний.

## Решение: Nikel — Knowledge Graph + Hybrid RAG для научных R&D

Nikel — это интеллектуальная система для Obsidian, которая автоматически строит **граф знаний** и **гибридный векторный индекс** из научно-технических документов (PDF, TXT, DOCX) и отвечает на сложные многопараметрические запросы на естественном языке.

### Ключевая идея

```
Разрозненные документы
  ├── → Vision LLM / pdfjs → Markdown
  ├── → Entity Extraction + Relation Mining → Knowledge Graph (графовый поиск)
  └── → Semantic Chunking + Embeddings → Vector Index (гибридный RAG: BM25 + cosine similarity)
        ↓
LLM-ассистент с контекстом из графа И/ИЛИ векторного поиска
```

## Технологический стек: почему это мощно

### 🔬 AI + NLP

| Технология | Роль в Nikel | Преимущество |
|------------|-------------|--------------|
| **Ollama** (Gemma 4, LLaVA) | Локальный LLM-сервер | Все данные остаются на вашем компьютере. Zero внешних API. Full privacy. Работает без интернета |
| **Ollama Embeddings API** (`/api/embed`) | Генерация векторных эмбеддингов для семантического поиска | Batch-mode: один запрос на массив текстов. Поддержка любых embedding-моделей (nomic-embed-text, all-minilm, bge-m3) |
| **Vision LLM** | Понимает PDF как изображение — читает сканы, схемы, таблицы, графики | Единственное решение, которое «видит» PDF целиком, а не только слой текста. Работает там, где pdfplumber и PyPDF2 бессильны |
| **Cosine Similarity** | Мера семантической близости между эмбеддингом запроса и чанков | a·b / (|a|·|b|). Нормализованное сравнение векторов — основа семантического поиска |
| **BM25 Okapi** | Классический ранжировщик по ключевым словам | k1=1.5, b=0.75 — золотой стандарт keyword search. Компенсирует то, что семантика может упустить редкие точные термины |
| **Гибридный поиск (BM25 + cosine)** | Взвешенная комбинация: 30% BM25 + 70% семантика | Лучшее из двух миров: точность ключевых слов + глубина семантики |
| **Semantic Chunking** | Деление текста по границам параграфов (\n\n) с overlap 200 | Чанки сохраняют смысловую целостность. Длинные параграфы режутся скользящим окном 1000/200 |
| **Query Rewriting** | LLM расширяет запрос пользователя: добавляет синонимы, английские термины, уточнения | «Режимы циркуляции» → «режимы циркуляции circulation modes flow patterns». Fallback на оригинал при ошибке |
| **Entity-Relation Extraction** | Извлечение 12 типов сущностей + 13 типов связей | Промышленная онтология: материалы, процессы, оборудование, свойства, режимы, выводы, публикации — со всеми метаданными |

### 🗄️ Граф знаний (Graph Mode)

| Технология | Роль | Преимущество |
|------------|------|-------------|
| **In-memory Knowledge Graph** | Хранение сущностей и связей с dedup и merge | Молниеносный поиск по графу < 1ms на запрос. Никаких внешних БД — всё в JSON внутри Obsidian |
| **Многоуровневая фильтрация** | SearchFilters: тип + география + год + confidence + числовые параметры + теги | Отвечает на запросы вида «найди российские публикации после 2020 года про выщелачивание никеля с концентрацией сульфатов < 200 мг/л» |
| **BFS Cluster** | Построение 1-hop и 2-hop окрестностей сущности | Мгновенно показывает полный контекст: «какие эксперименты, материалы, оборудование и выводы связаны с этой публикацией» |
| **Batch dedup + merge** | Слияние дубликатов по (type + normalizedName) | Гарантирует, что «Электроэкстракция никеля» и «electrowinning of nickel» — одна сущность, а не две |

### 🔎 Векторный поиск (Direct Mode + Hybrid RAG)

| Технология | Роль | Преимущество |
|------------|------|-------------|
| **TextChunk[] + Embeddings** | Каждый чанк текста хранит свой вектор эмбеддинга number[] | Возможность семантического сравнения без внешней векторной БД |
| **Batch Embedding Generation** | `ollama.getEmbeddings({ input: chunks[] })` — один HTTP-запрос на все чанки документа | Не N запросов на документ, а 1. Радикальное ускорение индексации |
| **TOP_K_RERANK=20** | Гибридный поиск отбирает 20 кандидатов, финальный topK=5 | Больше релевантных кандидатов на входе → выше качество финального ответа |
| **Keyword Fallback** | Если rewritten query не нашёл ничего → поиск по оригинальному запросу | Отказоустойчивость: LLM-расширение может ошибиться, оригинал — всегда запасной вариант |
| **Embedding On/Off** | Настройка `embeddingEnabled: boolean` | Без эмбеддингов — классический keyword search. Для слабых машин или когда не нужна семантика |

### 📄 Обработка документов

| Технология | Роль | Преимущество |
|------------|------|-------------|
| **PDF.js v4 (pdfjs-dist)** | Рендеринг PDF в PNG + извлечение текстового слоя | Единственная pure-JS библиотека, работающая внутри Obsidian. Не требует Python, Native-модулей или серверов |
| **Mammoth** (pure JS) | Конвертация DOCX → Markdown | Поддержка Microsoft Word документов без LibreOffice или COM-объектов |
| **TextDecoder** | TXT с автоопределением BOM | Корректная обработка UTF-8, UTF-16, файлов с BOM |
| **Promise.allSettled** | Отказоустойчивая обработка страниц PDF | Сбой на одной странице не убивает весь документ. PDF с битыми страницами обрабатывается полностью |
| **Escalating timeout** (90s→180s→360s) | Адаптивная стратегия для Vision LLM | Быстрая выдача результата на лёгких страницах, большой запас на сложных |

### ⚡ Архитектура и производительность

| Технология | Преимущество |
|------------|-------------|
| **Incremental indexing (MD5)** | При повторном запуске обрабатываются только новые/изменённые файлы. Индексация 1000 PDF → 2 секунды на проверку |
| **Debounced flush (200ms / 20 строк)** | Логирование без тормозов: накопление строк в памяти, групповая запись на диск |
| **Ring buffer (500 строк)** | Автоматическое управление размером лога — без разрастания |
| **Атомарные записи (write → .tmp → rename)** | Никаких повреждённых JSON. Если система упадёт в момент записи — файл останется целым |
| **TOCTOU-safe file watcher** | Защита от race condition: файл, удалённый между сканированием и расчётом хеша, просто пропускается |
| **Promise.allSettled batch** | Параллельная обработка до N страниц PDF с отказоустойчивостью |

### 🧠 Граф знаний: 12 типов сущностей, 13 типов связей

| Тип сущности | Что описывает | Примеры из металлургии |

| **Equipment** | Оборудование | Ванны электроэкстракции, печи взвешенной плавки, системы очистки газов |
| **Team** | Лаборатории, организации | Лаборатория гидрометаллургии, кафедра МИСиС |
| **Person** | Исследователи | Авторы публикаций, эксперты |
| **Conclusion** | Выводы, результаты | «Оптимальная скорость циркуляции — 1.2-1.8 м/с» |
| **Topic** | Темы, области | Обессоливание воды, переработка отходов, пирометаллургия |
| **Publication** | Статьи, патенты, отчёты | Литературные обзоры, патент РФ №..., диссертации |
| **Process** | Технологические процессы | Выщелачивание, электроэкстракция, плавка, циркуляция |
| **Facility** | Заводы, установки | Промышленные площадки, лабораторные стенды |

Каждая сущность хранит метаданные:
- **Confidence** — уровень достоверности (high/medium/low)
- **Geography** — географическая принадлежность (ru/foreign/both)
- **Year** — год публикации/эксперимента
- **SourceType** — тип источника (article/report/patent/conference/review/dissertation)
- **Properties** — числовые и текстовые характеристики с единицами измерения

### 🔍 Поиск и аналитика

Nikel предоставляет **два независимых поисковых движка**, которые можно использовать по отдельности или в комбинации:

#### Graph Mode — графовый поиск (Knowledge Graph)

Работает с сущностями и связями. Идеален для запросов вида «найди все эксперименты, в которых использовался Материал-X и получил Свойство-Y».

```
Запрос → LLM извлекает имена сущностей → графовый поиск по (тип + имя) → 
BFS-кластеризация 1-hop/2-hop → контекст со [[ссылками]] → LLM-ответ
```

#### Direct Mode — гибридный RAG (DocumentStore + Embeddings)

Работает с сырым текстом чанков. Идеален когда граф ещё не построен или нужен поиск по конкретным формулировкам.

```
Запрос → LLM-расширение (rewriting) → query embedding (cosine) + BM25 keywords →
гибридная сортировка → TOP_K_RERANK=20 → top-5 чанков → LLM-ответ с контекстом
```

Оба движка отвечают на запросы, невозможные в обычном поиске:

> **«Какие методы обессоливания воды подходят для сульфатов до 300 мг/л с сухим остатком ≤1000 мг/дм³?»**

> **«Покажи все эксперименты по распределению Au, Ag, МПГ между штейном и шлаком за последние 5 лет»**

> **«Какие решения циркуляции католита при электроэкстракции никеля описаны в мировой практике?»**

### 📊 Визуализация

- **Interactive Canvas** — граф знаний с grid-обзором, radial-кластерами и timeline-хронологией
- **Mermaid-диаграмма** — `_graph.md` с рендерингом прямо в Obsidian (LR-graph с легендой типов)
- **Dataview-поля** — каждая сущность = заметка с машиночитаемыми полями для Obsidian Dataview
- **Каталог** — `_index.md` со всеми сущностями, сгруппированными по типам

### 🔄 Инкрементальная работа

- **MD5-хеширование** — повторная индексация обрабатывает только новые/изменённые файлы
- **Graceful degradation** — сбой страницы PDF = пустая строка (не крах)
- **Entity dedup** — дубликаты сущностей автоматически мержатся по (type + normalizedName)
- **Relation dedup** — дубликаты связей отсеиваются по (from + to + type)

### 🛡️ Надёжность и отказоустойчивость

- **Транзакционная запись:** все JSON-файлы пишутся через write → .tmp → rename
- **Auto-backup:** при повреждении `index.json` создаётся `.bak`-копия
- **Error isolation:** ошибка одного файла не прерывает индексацию остальных
- **TOCTOU-safe:** защита от race condition при файловых операциях
- **Timeout escalation:** если Vision LLM подвисла — 90s → 180s → 360s adaptive timeout

### 🎛️ Режимы использования

| Сценарий | Режим | Что происходит |
|----------|-------|----------------|
| У вас есть PDF со сканами и схемами | `Vision` | Каждая страница → PNG → Vision LLM. Распознаёт даже рукописные пометки |
| У вас тысячи текстовых статей | `Fast` | getTextContent за миллисекунды. Vision только для страниц без текста |
| Нужен быстрый поиск без графа | `Direct` | Извлечение текста → семантический чанкинг → гибридный поиск (BM25 + cosine). Эмбеддинги опционально |
| Уже построили граф | Graph mode | @nikel_s ищет по графу + отвечает с контекстом и [[ссылками]] |
| Нет источников | Direct LLM | @nikel_s работает как обычный AI-ассистент без контекста |

## Три режима индексации: полный, быстрый, прямой

Nikel предлагает **три принципиально разных режима индексации**, каждый со своей архитектурой, технологиями и областью применения. Это не просто «настройки скорости» — это три разных подхода к извлечению знаний.

### 🏛️ Полный режим (Vision) — максимальное качество

**Для кого:** У вас есть PDF со сканами, схемами, рукописными пометками, фотографиями страниц. Вам нужно извлечь максимум информации — включая сущности и связи для графа знаний.

**Архитектура:**
```
PDF-файл → pdfjs-dist v4 (200 DPI рендер PNG) → 
  для каждой страницы: Vision LLM (Gemma 4 / LLaVA) с изображением →
  Markdown-транскрипция →
EntityExtractor (LLM → JSON): 12 типов сущностей + 13 типов связей →
KnowledgeGraph (dedup + merge + транзакционный JSON)
```

**Технологии:**
- **PDF.js v4** — рендеринг каждой страницы в PNG для Vision LLM
- **Vision LLM** (через Ollama `/api/chat` с `images[]`) — распознаёт текст, таблицы, схемы, графики, рукописные пометки
- **Escalating timeout** (90s → 180s → 360s) — адаптивная стратегия: быстрая выдача на лёгких страницах, большой запас на сложных
- **EntityExtractor + Prompt Engineering** — кастомные промпты для извлечения металлургической онтологии
- **KnowledgeGraph** — in-memory граф с BFS-кластеризацией и многоуровневой фильтрацией

**Скорость:** ~5-15 секунд на страницу (зависит от модели и сложности страницы).

**Когда выбирать:**
- PDF содержит сканы, схемы, графики, рукописный текст
- Вам нужен Knowledge Graph с сущностями и связями
- Качество важнее скорости

**Почему это отличное решение:** Единственный режим, который «видит» документ целиком через Vision LLM. Традиционные подходы (pdfplumber, PyPDF2) извлекают только текстовый слой — они бессильны против сканов, схем, фотографий страниц. Vision LLM понимает контекст: таблица, график, формула — и возвращает структурированный Markdown, который EntityExtractor превращает в граф знаний.

---

### ⚡ Быстрый режим (Fast) — скорость + Quality of Life

**Для кого:** У вас преимущественно текстовые PDF (статьи, отчёты, патенты), но иногда попадаются страницы со схемами или таблицами. Вам нужен граф знаний, но вы не готовы ждать Vision LLM на каждую страницу.

**Архитектура:**
```
PDF-файл → pdfjs-dist v4 (извлечение текстового слоя через getTextContent) →
  если текст страницы ≥ 200 символов → Markdown (мгновенно)
  если текст < 200 символов → Vision LLM fallback (PNG → распознавание)
  если getTextContent упал → Vision LLM fallback
→ Aggregation (LLM для >3 страниц, raw join для ≤3) →
EntityExtractor (LLM → JSON) → KnowledgeGraph
```

**Технологии:**
- **PDF.js getTextContent** — извлечение текстового слоя за миллисекунды (без рендеринга PNG)
- **Threshold 200 символов** — если текстовый слой достаточен, Vision не вызывается
- **Graceful fallback** — если getTextContent вернул <200 символов или упал, страница переходит на Vision
- **Aggregation LLM** — для документов >3 страниц: LLM собирает Markdown-страницы в единый связный документ

**Скорость:** миллисекунды для текстовых страниц, ~5-15 сек для страниц-исключений.

**Когда выбирать:**
- У вас тысячи текстовых PDF
- Некоторые страницы содержат схемы/таблицы/рисунки
- Вам нужен граф знаний, но с приемлемым временем индексации

**Почему это отличное решение:** Умная оптимизация — 95% страниц обрабатываются за миллисекунды, оставшиеся 5% (схемы, таблицы, сканы) получают полную Vision-обработку. Вы получаете граф знаний почти так же быстро, как в Direct режиме, но с Quality of Life Vision для сложных страниц. Это лучший баланс для реальных научных архивов.

### 🧠 Прямой режим (Direct) — гибридный RAG без графа

**Для кого:** Вам нужен быстрый поиск по тексту документов без построения графа знаний. Или у вас нет подходящей Vision-модели. Direct режим + Embeddings = полноценный RAG-пайплайн.

**Архитектура:**
```
PDF-файл → pdfjs-dist getTextContent (fast mode) →
  Semantic Chunking (по параграфам \n\n, overlap 200 для длинных) →
  [Опционально] Batch Embeddings через Ollama /api/embed →
    DocumentStore.addChunk(text, embedding) для каждого чанка
  [Без эмбеддингов] DocumentStore.addDocument(text) → внутренний semanticChunk
  
Поиск:
  Запрос → Query Rewriting (LLM: синонимы + английские термины) →
  [С эмбеддингами] ollama.getEmbeddings({ input: rewrittenQuery }) →
    Гибридный поиск: BM25(0.3) + cosine(0.7) → TOP_K_RERANK=20 → top-5
  [Без эмбеддингов] Keyword search (case-insensitive .includes)
  Если rewritten query не нашёл ничего → fallback на оригинальный запрос
  → LLM-ответ с контекстом найденных чанков
```

**Технологии:**
- **Semantic Chunking** — деление текста по границам параграфов (`\n\n`), а не по фиксированной длине. Длинные параграфы (>1000 символов) режутся скользящим окном с overlap 200. Чанки сохраняют смысловую целостность
- **Ollama `/api/embed`** — batch-генерация эмбеддингов: один HTTP-запрос на массив чанков. Поддерживает любые embedding-модели (nomic-embed-text, all-minilm, bge-m3). Векторы хранятся прямо в TextChunk.embeddings — никакой внешней векторной БД
- **Cosine Similarity** — мера близости между query embedding и chunk embedding. `a·b / (|a|·|b|)`. Возвращает 0..1
- **BM25 Okapi** (k1=1.5, b=0.75) — классический keyword scorer. IDF через `ln(1 + (N - df + 0.5) / (df + 0.5))`. Работает как fallback, когда семантика не находит точное ключевое слово
- **Гибридный поиск** — `score = 0.3 * BM25 + 0.7 * cosine`. Веса настроены так, что семантика доминирует, но редкие точные термины (например, «шпургельт») не теряются
- **TOP_K_RERANK=20** — сначала отбирает 20 кандидатов по гибридному скору, затем отсекает top-5 по финальной сортировке. Больше кандидатов на входе → выше качество
- **Query Rewriting** — LLM преобразует запрос пользователя в улучшенную версию: «режимы циркуляции католита» → «режимы циркуляции католита circulation modes catholyte flow patterns electrolyte movement». Fallback на оригинал при ошибке LLM
- **Утилиты: tokenize()** — unicode-aware токенизация (`\p{L}+`), корректно работает с кириллицей и латиницей

**Скорость:** секунды на индексацию (даже с эмбеддингами — один batch-запрос на документ). Поиск — десятки миллисекунд.

**Когда выбирать:**
- Вам нужен быстрый поиск без графа знаний
- У вас нет Vision-модели или вы не хотите ждать
- Вы хотите RAG-пайплайн с семантическим поиском
- У вас слабая машина — эмбеддинги можно отключить, оставив keyword search

**Почему это отличное решение:** Единственный режим, который предоставляет **полноценный гибридный RAG** (BM25 + cosine similarity + query rewriting) внутри Obsidian без единой внешней зависимости. Никакой векторной БД, никаких облачных API — только локальный Ollama и чистый TypeScript. При этом:
- **BM25** гарантирует, что точные технические термины не потеряются
- **Cosine similarity** находит семантически близкие тексты, даже если в них нет ни одного общего слова
- **Query rewriting** улучшает запрос перед поиском и имеет fallback на оригинал
- **Embeddings On/Off** — гибкая настройка: включите для семантического поиска, отключите для максимальной скорости на слабых машинах

| Характеристика | Vision (Полный) | Fast (Быстрый) | Direct (Прямой) |

| **Поиск** | Knowledge Graph | Knowledge Graph | Гибридный RAG (BM25 + cosine) |

## Быстрый старт

```bash
# 1. Установите Ollama
curl -fsSL https://ollama.com/install.sh | sh
ollama serve

# 2. Загрузите модель (хотя бы одну)
ollama pull gemma4:e4b       # для ответов и Vision

# 3. (Опционально) Загрузите модель эмбеддингов для гибридного RAG
ollama pull nomic-embed-text

# 4. Скопируйте плагин в Obsidian
cp -r nikel-plugin ~/ваш-вault/.obsidian/plugins/

# 5. Включите в Obsidian: Настройки → Сторонние плагины → Nikel
```

### Первая индексация

1. Укажите папку с источниками в настройках Nikel
2. Выберите режим: `Vision` (максимальное качество) / `Fast` (максимальная скорость) / `Direct` (RAG без графа)
3. (Direct) Включите эмбеддинги и укажите модель в настройках
4. Запустите: `Cmd+P` → **Nikel: Индексировать папки источников**
5. Готово! Откройте `nikel/_index.md` и `nikel/canvas/обзор-базы-знаний.canvas`

### Использование

В любом документе Obsidian:
```
@nikel_s
Какие методы обессоливания воды применялись для сульфатов до 300 мг/л?
```

Запустите команду **Process @nikel task** → Nikel найдёт ответ (в графе или векторном индексе) и вставит его в документ.

## Сравнение: Graph Mode vs Direct Mode (Hybrid RAG)

| Аспект | Graph Mode | Direct Mode |
|--------|-----------|-------------|
| **Что ищет** | Сущности и их связи | Сырой текст чанков |
| **Как хранит** | 12 типов + 13 связей в JSON | TextChunk[] с эмбеддингами |
| **Поиск** | LLM → Extract → BFS → Rerank | BM25(0.3) + Cosine(0.7) → Rerank |
| **Когда строить** | Vision/Fast индексация | Direct индексация |
| **Сильная сторона** | «Что связано с чем?» (графовые запросы) | «Где написано про X?» (поиск по тексту) |
| **Слабая сторона** | Не ищет по сырому тексту | Нет графовой аналитики |
| **LLM на индексацию** | Vision + EntityExtraction | Embeddings (опционально) |
| **LLM на поиск** | EntityExtraction + answer | Query rewriting + answer |
