# Nikel — Интеллектуальная карта знаний для горно-металлургических исследований

> **Развяжите «Научный клубок»** — превратите разрозненные отчёты, статьи и эксперименты в единую, живую базу знаний с графовой аналитикой и AI-ассистентом.

---

## Проблема: Почему ценные знания теряются

В горно-металлургических исследованиях знания рассеяны по тысячам документов:

- **Потеря институциональной памяти** — методы обессоливания воды, режимы циркуляции католита, распределение металлов между штейном и шлаком погребены в отчётах и личных архивах
- **Дублирование усилий** — команды повторяют литературные обзоры, не видя, что работа уже выполнена
- **Междисциплинарная слепота** — невозможно быстро найти связь «режим кучного выщелачивания в холодном климате → выход металла» когда данные разрознены
- **Медленные решения** — ответ на вопрос «какие решения циркуляции электролита существуют?» требует ручного сбора из десятков источников
- **Риск противоречий** — без единой верифицированной базы множатся конфликты интерпретаций

**Nikel** — решение, которое превращает этот хаос в управляемую, семантически связанную карту знаний.

---

## Решение: Nikel — Knowledge Graph + LLM для научных R&D

Nikel — это интеллектуальная система для Obsidian, которая автоматически строит **граф знаний** из научно-технических документов (PDF, TXT, DOCX) и отвечает на сложные многопараметрические запросы на естественном языке.

### Ключевая идея

```
Разрозненные документы → Entity Extraction + Relation Mining → Knowledge Graph → LLM-ассистент с контекстом
```

## Технологический стек: почему это мощно

### 🔬 AI + NLP

| Технология | Роль в Nikel | Преимущество |
|------------|-------------|--------------|
| **Ollama** (Gemma 4, LLaVA) | Локальный LLM-сервер | Все данные остаются на вашем компьютере. Zero внешних API. Full privacy. Работает без интернета |
| **Vision LLM** | Понимает PDF как изображение — читает сканы, схемы, таблицы, графики | Единственное решение, которое «видит» PDF целиком, а не только слой текста. Работает там, где `pdfplumber` и `PyPDF2` бессильны |
| **Prompt Engineering** (кастомные промпты) | Управляет извлечением сущностей, агрегацией страниц, ответами | Полный контроль над онтологией предметной области. Адаптация под металлургическую терминологию |
| **Entity-Relation Extraction** | Извлечение 12 типов сущностей + 13 типов связей | Промышленная онтология: материалы, процессы, оборудование, свойства, режимы, выводы, публикации — со всеми метаданными |

### 🗄️ Граф знаний

| Технология | Роль | Преимущество |
|------------|------|-------------|
| **In-memory Knowledge Graph** | Хранение сущностей и связей с dedup и merge | Молниеносный поиск по графу < 1ms на запрос. Никаких внешних БД — всё в JSON внутри Obsidian |
| **Многоуровневая фильтрация** | SearchFilters: тип + география + год + confidence + числовые параметры + теги | Отвечает на запросы вида «найди российские публикации после 2020 года про выщелачивание никеля с концентрацией сульфатов < 200 мг/л» |
| **BFS Cluster** | Построение 1-hop и 2-hop окрестностей сущности | Мгновенно показывает полный контекст: «какие эксперименты, материалы, оборудование и выводы связаны с этой публикацией» |
| **Batch dedup + merge** | Слияние дубликатов по (type + normalizedName) | Гарантирует, что «Электроэкстракция никеля» и «electrowinning of nickel» — одна сущность, а не две |

### 📄 Обработка документов

| Технология | Роль | Преимущество |
|------------|------|-------------|
| **PDF.js v4 (pdfjs-dist)** | Рендеринг PDF в PNG + извлечение текстового слоя | Единственная pure-JS библиотека, работающая внутри Obsidian. Не требует Python, Native-модулей или серверов |
| **Mammoth** (pure JS) | Конвертация DOCX → Markdown | Поддержка Microsoft Word документов без LibreOffice или COM-объектов |
| **TextDecoder** | TXT с автоопределением BOM | Корректная обработка UTF-8, UTF-16, файлов с BOM |
| **Promise.allSettled** | Отказоустойчивая обработка страниц PDF | Сбой на одной странице не убивает весь документ. PDF с битыми страницами обрабатывается полностью |
| **Escalating timeout** (90s→180s→360s) | Адаптивная стратегия для Vision LLM | Быстрая выдача результата на лёгких страницах, большой запас на сложных |

### ⚡ Архитектура и производительность

| Технология | Преимущество |
|------------|-------------|
| **Incremental indexing (MD5)** | При повторном запуске обрабатываются только новые/изменённые файлы. Индексация 1000 PDF → 2 секунды на проверку |
| **Debounced flush (200ms / 20 строк)** | Логирование без тормозов: накопление строк в памяти, групповая запись на диск |
| **Ring buffer (500 строк)** | Автоматическое управление размером лога — без разрастания |
| **Атомарные записи (write → .tmp → rename)** | Никаких повреждённых JSON. Если система упадёт в момент записи — файл останется целым |
| **TOCTOU-safe file watcher** | Защита от race condition: файл, удалённый между сканированием и расчётом хеша, просто пропускается |
| **Promise.allSettled batch** | Параллельная обработка до N страниц PDF с отказоустойчивостью |

### 🎨 Генерация и визуализация

| Технология | Что создаёт |
|------------|-------------|
| **MdGenerator** | `.md`-заметки с Dataview-полями для каждой сущности. Совместимость с Obsidian Dataview, Graph View, Backlinks |
| **CanvasGenerator** | Интерактивные `.canvas`-графы: обзор базы знаний (grid), кластеры (radial), хронология (timeline) |
| **IndexGenerator** | `_index.md` (каталог по типам), `_graph.md` (Mermaid-диаграмма) |
| **YAML-frontmatter** | Ручной сериализатор YAML без внешних зависимостей. Dataview-совместимые поля для всех 13 типов связей |

### 🔐 Безопасность и приватность

| Аспект | Как решено |
|--------|-----------|
| **Данные** | Всё хранится локально в Obsidian vault. Никакие документы не отправляются на внешние серверы |
| **LLM** | Локальный Ollama. Даже Vision-обработка PDF идёт через вашу машину |
| **Сеть** | Zero external API calls. Единственный сетевой запрос — к localhost:11434 (Ollama) |
| **Аудит** | Полный лог всех действий с таймстемпами: индексация, запросы, ошибки. Экспорт лога одной кнопкой |

## Возможности: что умеет Nikel

### 📥 Импорт и нормализация данных

Поддерживаемые форматы:
- **PDF** — Vision + текстовый слой. Работает со сканами, фотографиями страниц, схемами, таблицами
- **DOCX** — через Mammoth (Microsoft Word)
- **TXT** — любые кодировки с автоопределением BOM

Три режима индексации:

| Режим | Скорость | Использование LLM | Для чего |
|-------|----------|-------------------|----------|
| **Vision** | ~5-15 с/стр | Vision LLM на каждую страницу | Сканы, схемы, таблицы, любые PDF |
| **Fast** | ~мс/стр | Vision только если текст <200 символов | Текстовые PDF (статьи, отчёты) |
| **Direct** | ~с/файл | Без LLM (только keyword search) | Быстрый поиск без графа знаний |

### 🧠 Граф знаний: 12 типов сущностей, 13 типов связей

| Тип сущности | Что описывает | Примеры из металлургии |
|:---|---:|:---|
| **Material** | Материалы, сплавы, составы | Никель, сульфаты, техногенный гипс, угольные отходы |
| **Experiment** | Эксперименты, протоколы | Кучное выщелачивание в холодном климате, электроэкстракция при 60°C |
| **Property** | Свойства, характеристики | Прочность, концентрация (мг/л), температура плавления |
| **Mode** | Режимы, параметры | Давление 2 атм, скорость потока 1.5 м/с, pH 4.5 |
| **Equipment** | Оборудование | Ванны электроэкстракции, печи взвешенной плавки, системы очистки газов |
| **Team** | Лаборатории, организации | Лаборатория гидрометаллургии, кафедра МИСиС |
| **Person** | Исследователи | Авторы публикаций, эксперты |
| **Conclusion** | Выводы, результаты | «Оптимальная скорость циркуляции — 1.2-1.8 м/с» |
| **Topic** | Темы, области | Обессоливание воды, переработка отходов, пирометаллургия |
| **Publication** | Статьи, патенты, отчёты | Литературные обзоры, патент РФ №..., диссертации |
| **Process** | Технологические процессы | Выщелачивание, электроэкстракция, плавка, циркуляция |
| **Facility** | Заводы, установки | Промышленные площадки, лабораторные стенды |

Каждая сущность хранит метаданные:
- **Confidence** — уровень достоверности (high/medium/low)
- **Geography** — географическая принадлежность (ru/foreign/both)
- **Year** — год публикации/эксперимента
- **SourceType** — тип источника (article/report/patent/conference/review/dissertation)
- **Properties** — числовые и текстовые характеристики с единицами измерения

### 🔍 Поиск и аналитика: отвечает на сложные вопросы

Nikel отвечает на запросы, которые невозможны в обычном поиске:

> **«Какие методы обессоливания воды подходят для сульфатов до 300 мг/л с сухим остатком ≤1000 мг/дм³?»**

Система:
1. Извлекает ключевые сущности из вопроса (методы обессоливания, сульфаты, диапазоны)
2. Ищет связанные сущности в графе (материалы → процессы → публикации)
3. Фильтрует по числовым параметрам (сульфаты ≤ 300 мг/л)
4. Генерирует ответ с [[ссылками на источники]]

> **«Покажи все эксперименты по распределению Au, Ag, МПГ между штейном и шлаком за последние 5 лет»**

> **«Какие решения циркуляции католита при электроэкстракции никеля описаны в мировой практике?»**

> **«Какие российские исследования по очистке шахтных вод существуют, и какие команды этим занимались?»**

### 📊 Визуализация

- **Interactive Canvas** — граф знаний с grid-обзором, radial-кластерами и timeline-хронологией
- **Mermaid-диаграмма** — `_graph.md` с рендерингом прямо в Obsidian (LR-graph с легендой типов)
- **Dataview-поля** — каждая сущность = заметка с машиночитаемыми полями для Obsidian Dataview
- **Каталог** — `_index.md` со всеми сущностями, сгруппированными по типам

### 🔄 Инкрементальная работа

- **MD5-хеширование** — повторная индексация обрабатывает только новые/изменённые файлы
- **Graceful degradation** — сбой страницы PDF = пустая строка (не крах)
- **Entity dedup** — дубликаты сущностей автоматически мержатся по (type + normalizedName)
- **Relation dedup** — дубликаты связей отсеиваются по (from + to + type)

### 🛡️ Надёжность и отказоустойчивость

- **Транзакционная запись:** все JSON-файлы пишутся через write → .tmp → rename
- **Auto-backup:** при повреждении `index.json` создаётся `.bak`-копия
- **Error isolation:** ошибка одного файла не прерывает индексацию остальных
- **TOCTOU-safe:** защита от race condition при файловых операциях
- **Timeout escalation:** если Vision LLM подвисла — 90s → 180s → 360s adaptive timeout

### 🎛️ Режимы использования

| Сценарий | Режим | Что происходит |
|----------|-------|----------------|
| У вас есть PDF со сканами и схемами | `Vision` | Каждая страница → PNG → Vision LLM. Распознаёт даже рукописные пометки |
| У вас тысячи текстовых статей | `Fast` | getTextContent за миллисекунды. Vision только для страниц без текста |
| Нужен быстрый поиск без графа | `Direct` | Только извлечение текста + keyword search. Секунды на всю папку |
| Уже построили граф | Graph mode | @nikel_s ищет по графу + отвечает с контекстом и [[ссылками]] |
| Нет источников | Direct LLM | @nikel_s работает как обычный AI-ассистент без контекста |

## Быстрый старт

```bash
# 1. Установите Ollama
curl -fsSL https://ollama.com/install.sh | sh
ollama serve

# 2. Загрузите модель
ollama pull gemma4:e4b

# 3. Скопируйте плагин в Obsidian
cp -r nikel-plugin ~/ваш-вault/.obsidian/plugins/

# 4. Включите в Obsidian: Настройки → Сторонние плагины → Nikel
```

### Первая индексация

1. Укажите папку с источниками в настройках Nikel
2. Выберите режим: `Vision` (максимальное качество) / `Fast` (максимальная скорость) / `Direct` (без LLM)
3. Запустите: `Cmd+P` → **Nikel: Индексировать папки источников**
4. Готово! Откройте `nikel/_index.md` и `nikel/canvas/обзор-базы-знаний.canvas`

### Использование

В любом документе Obsidian:
```
@nikel_s
Какие методы обессоливания воды применялись для сульфатов до 300 мг/л?
```

Запустите команду **Process @nikel task** → Nikel найдёт ответ в графе и вставит его в документ.
